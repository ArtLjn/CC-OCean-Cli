# 记忆系统使用教程

> Ocean CLI 拥有双层记忆架构：手动记忆（`/mem` 命令）和自动记忆（extractMemories）。本教程详解两种记忆的原理和使用方法。

---

## 双层记忆架构概述

```
┌──────────────────────────────────────────────────────┐
│  手动记忆层 — /mem 命令                                │
│  用户主动保存，精确控制内容                              │
│  存储：<project>/.claude/memory/chunks/               │
│  索引：<project>/.claude/memory/index.json            │
├──────────────────────────────────────────────────────┤
│  自动记忆层 — extractMemories                         │
│  每次对话结束后后台自动提取值得记住的信息                  │
│  存储：<project>/.claude/memory/                      │
│  整合：AutoDream 定期去重和修剪                         │
└──────────────────────────────────────────────────────┘
```

两层系统互不冲突：
- **手动记忆**：用户用 `/mem` 命令显式保存，内容精确
- **自动记忆**：模型在对话结束后后台分叉代理提取，覆盖用户偏好、项目状态等

两层的记忆都会在后续对话中注入到系统提示中，为模型提供上下文。

## /mem 命令完整用法

`/mem` 是手动记忆管理的核心命令，提供 6 个子命令。

### 查看所有记忆

```
> /mem list
> /mem ls
```

列出当前项目所有已保存的记忆条目，显示 ID、标题和创建时间。

### 添加记忆

**普通模式**（LLM 自动总结为 100-200 字）：

```
> /mem add 用户偏好使用 PostgreSQL 而非 MySQL
> /mem add 本项目的认证中间件使用 JWT，token 有效期 24 小时
```

**`--full` 模式**（保存完整 Markdown 交接文档，适合复杂上下文）：

```
> /mem add --full 本次重构的完整决策记录和注意事项
```

在 `--full` 模式下，你可以输入完整的 Markdown 内容作为交接文档，不会被自动压缩。适合保存：

- 架构决策记录
- 复杂的上下文交接信息
- 详细的项目配置说明

### 查看记忆详情

```
> /mem show <memory-id>
```

显示指定记忆的完整内容。

### 搜索记忆

```
> /mem search 数据库
> /mem search authentication middleware
```

在记忆库中搜索关键词，返回匹配的记忆条目。

### 删除记忆

```
> /mem rm <memory-id>
```

删除指定的记忆条目。

### 帮助

```
> /mem help
```

显示 `/mem` 命令的使用帮助。

## 自动记忆提取的工作原理

### 触发时机

每次模型回复完成（且无 tool_use 需要继续执行）后，Ocean CLI 会在后台启动一个分叉代理（fire-and-forget），分析本次对话是否包含值得记住的信息。

### 提取流程

```
用户对话结束
  → 模型返回最终回复
  → 后台分叉代理启动
  → 代理分析对话内容
  → 决定是否有值得记录的内容
  → 如有，保存为 Markdown 文件到 .claude/memory/
  → 如无，静默退出
```

### 提取内容范围

自动提取关注以下类型的信息：

- **用户偏好**：工作风格、技术偏好、沟通方式
- **项目状态**：正在进行的工作、已做的决策、重要变更
- **反馈信息**：用户对模型行为的纠正和肯定
- **外部参考**：外部系统链接、文档位置

### 排除内容

以下内容**不会**被自动提取（即使出现也不会保存）：

- 可从代码直接推导的信息（代码模式、架构、文件路径）
- Git 历史或 recent changes
- 临时状态和调试方案
- 已记录在 CLAUDE.md 中的信息

### 控制自动提取

通过环境变量禁用自动记忆：

```bash
CLAUDE_CODE_DISABLE_AUTO_MEMORY=1 ocean
```

或在 `settings.json` 中配置：

```json
{
  "autoMemoryEnabled": false
}
```

## AutoDream 整合机制

### 什么是 AutoDream

AutoDream 是记忆系统的后台整合服务，定期对积累的记忆进行去重、修剪和整理。

### 触发条件

- 每 **24 小时**自动运行一次
- 每积累 **5 个新会话**后触发

### 工作流程

```
积累的原始记忆文件
  → AutoDream 后台代理启动
  → 分析所有记忆，识别重复和过时内容
  → 合并相似记忆
  → 移除不再相关的内容
  → 更新 MEMORY.md 索引
```

### 关闭 AutoDream

在 `settings.json` 中设置：

```json
{
  "autoDreamEnabled": false
}
```

## 记忆类型说明

Ocean CLI 的记忆系统定义了 4 种记忆类型：

### user（用户记忆）

保存用户的角色、偏好、技术背景等个人信息。

**何时保存**：当了解到用户的角色、偏好、技术背景时

**示例**：
```
用户是数据科学家，当前专注于可观测性/日志方向
用户有 10 年 Go 经验，首次接触 React 前端
```

### feedback（反馈记忆）

保存用户对模型行为的指导——包括纠正和肯定。

**何时保存**：用户纠正模型行为（"不要这样做"）或确认某种做法有效（"做得好，保持"）

**结构**：
```
规则本身
**Why:** 用户给出的原因
**How to apply:** 何时/何地适用
```

**示例**：
```
集成测试必须使用真实数据库而非 mock
**Why:** 上季度出现过 mock 测试通过但生产迁移失败的事故
**How to apply:** 所有涉及数据库的测试用例
```

### project（项目记忆）

保存项目的当前状态、进行中的工作、重要决策等。

**何时保存**：了解到谁在做什么、为什么做、何时完成时

**结构**：
```
事实或决策
**Why:** 动机和原因
**How to apply:** 对后续工作的影响
```

**示例**：
```
auth 中间件重写是因为合规要求，不是技术债清理
**Why:** 法律部门标记了 session token 存储方式不合规
**How to apply:** 范围决策应优先合规而非人体工学
```

### reference（参考记忆）

保存指向外部系统的链接和位置信息。

**何时保存**：了解到外部系统资源的位置和用途时

**示例**：
```
Pipeline bug 追踪在 Linear 项目 "INGEST" 中
Grafana 面板 grafana.internal/d/api-latency 是 oncall 监控页面
```

## 记忆文件位置和结构

### 目录结构

```
<project-root>/.claude/memory/
├── MEMORY.md          # 自动记忆索引（AutoDream 维护）
├── index.json         # 手动记忆索引（/mem 命令维护）
└── chunks/            # 手动记忆内容文件
    ├── auth-refactor_001.md
    ├── db-preference_002.md
    └── ...
```

### 手动记忆文件格式

每个手动记忆文件是一个 Markdown 文件，包含 frontmatter：

```markdown
---
name: 记忆标题
description: 一句话描述
type: user
---

记忆正文内容...
```

### 自动记忆文件格式

自动提取的记忆也使用 Markdown 格式，带有类似的 frontmatter：

```markdown
---
name: 用户技术背景
description: 用户是数据科学家，专注可观测性方向
type: user
---

用户是数据科学家，目前在调查日志系统的覆盖情况。
有丰富的 Python 经验但对前端框架不太熟悉。
```

### 记忆注入机制

每次新对话开始时，Ocean CLI 会：

1. 读取 `MEMORY.md` 索引，获取最近 10 条记忆
2. 注入到系统提示中（总长度不超过 2000 字符）
3. 模型据此调整自己的行为和回复风格

## 最佳实践

1. **重要信息用 `/mem add`**：对于确定需要记住的信息，主动使用 `/mem` 保存，不要完全依赖自动提取
2. **复杂上下文用 `--full`**：涉及多步骤决策的上下文，使用 `--full` 模式保存完整文档
3. **定期清理**：用 `/mem list` 查看记忆，删除过时的条目
4. **善用搜索**：用 `/mem search` 快速定位相关记忆
5. **信任但验证**：记忆可能过时，对记忆中的重要事实，模型会自动验证当前状态
